# Next.js를 기본값으로 두는 것에 대한 반론

작성: 2026-06-15
유형: 기술 선택 반론 노트

"특별한 이유가 없으면 Next.js"라는 기본값(default) 전략에 반대한다. 기본 스택을 미리 못 박는 것 자체가, 그리고 그 기본값이 하필 Next.js인 것이 둘 다 문제다.

## 1. "기본값으로 고민을 줄인다"가 오히려 잘못된 선택을 고착시킨다

기본 스택을 정해두면 매 프로젝트의 요구사항을 진지하게 따지지 않게 된다. 고민을 줄이는 게 장점이 아니라, **선택을 회피하는 습관**이 된다. 정적 블로그에도, 단순 랜딩 페이지에도 Next.js를 얹는 과잉 엔지니어링이 흔하다. "기본값"은 사고 정지를 정당화한다.

## 2. Next.js는 풀스택으로 쓰기에 무겁고 락인이 강하다

- App Router·RSC·Server Actions는 학습 곡선이 가파르고 자주 바뀐다. 안정성을 원하는 팀에 부담.
- 백엔드 로직을 Next.js 안에 두면 프레임워크에 깊게 결합된다. 나중에 백엔드를 분리하려면 재작성 비용이 크다. "점진적 분리"는 말처럼 쉽지 않다.
- Vercel 생태계 락인. 셀프호스팅 시 일부 기능(ISR, 이미지 최적화, 엣지)이 까다로워진다.
- 풀스택이 필요 없는 프로젝트엔 순수 SPA(Vite+React) 또는 전용 백엔드가 더 단순하고 빠르다.

## 3. 코딩 에이전트 논거는 양날의 검이다

"예시가 풍부해서 에이전트가 잘 짠다"는 주장에는 반대 측면이 있다.

- Next.js는 **버전마다 권장 패턴이 크게 바뀌었다**(Pages Router → App Router). 그래서 웹·학습데이터에 *구식 예시가 대량으로 섞여 있다.* 에이전트가 옛 패턴(getServerSideProps 등)과 새 패턴(RSC)을 뒤섞어 환각하는 경우가 잦다.
- 예시가 많다는 건 *모순되는 예시도 많다*는 뜻이다. 에이전트 입장에서 "풍부함"이 곧 "정확함"은 아니다.
- 오히려 API 표면이 작고 안정적인 프레임워크(예: SvelteKit, Remix의 일부 버전)가 에이전트에게 더 일관된 코드를 내게 할 수 있다.

## 결론

기본 스택을 둘 거라면 프로젝트 유형별로 여러 기본값을 두거나, 매번 가볍게라도 재평가하는 게 낫다. Next.js는 "풀스택이 정말 필요하고, 최신 패턴을 따라갈 여력이 있는" 프로젝트의 강력한 후보지, 무조건적 기본값은 아니다.
