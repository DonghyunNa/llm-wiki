# 왜 Next.js를 기본 스택으로 쓰는가

작성: 2026-06-15
유형: 기술 선택 노트 (팀/개인 기본 스택 결정)

새 웹 프로젝트를 시작할 때 프레임워크 선택에서 매번 고민하지 않기 위해, **Next.js를 기본값(default)으로 둔다.** 특별한 이유가 없는 한 Next.js로 시작한다. 그 근거를 정리한다.

## 1. 풀스택 프레임워크로서 적절하다

Next.js는 프론트엔드(React 기반 UI)와 백엔드(API Routes / Route Handlers, Server Actions)를 한 코드베이스에서 다룰 수 있다. 별도의 백엔드 서버를 세우지 않고도:

- API 엔드포인트를 같은 프로젝트 안에 둘 수 있고,
- 서버 컴포넌트(RSC)로 데이터 페칭을 서버에서 처리하며,
- 인증·세션·DB 접근 같은 서버 로직을 프론트와 같은 레포에서 관리한다.

작은 팀이나 1인 개발에서 "프론트 레포 / 백엔드 레포"를 따로 운영하는 부담을 줄여준다. MVP를 빠르게 세우고, 필요해지면 백엔드를 분리하는 점진적 경로가 자연스럽다.

## 2. 예시(레퍼런스)가 풍부하다

Next.js는 React 생태계에서 사실상 표준 메타프레임워크 위치라, 문서·튜토리얼·오픈소스 예제·Stack Overflow 답변·블로그 글이 압도적으로 많다. 인증(NextAuth/Auth.js), 결제(Stripe), DB(Prisma/Drizzle), 배포(Vercel) 등 흔한 조합마다 검증된 예시가 존재한다.

## 3. 코딩 에이전트 시대에 "예시의 풍부함"은 결정적 장점이다

이게 핵심 논거다. Claude Code 같은 coding agent가 개발의 큰 부분을 맡는 지금, 프레임워크 선택 기준이 바뀐다.

- 코딩 에이전트는 학습 데이터·웹 검색·기존 예시 패턴에 기대어 코드를 생성한다.
- **예시가 풍부한 프레임워크일수록 에이전트가 더 정확하고 관용적인(idiomatic) 코드를 낸다.** 흔한 패턴일수록 에이전트가 헛디딜 확률이 낮다.
- 반대로 니치한 프레임워크는 에이전트가 자료 부족으로 환각하거나 구식 API를 쓰기 쉽다.

즉 "사람에게 익숙한 프레임워크"를 넘어 "에이전트가 잘 다루는 프레임워크"가 새 선택 축이 되었고, Next.js는 이 축에서 가장 앞선다. 에이전트와의 생산성이 곧 팀 생산성이 되는 흐름에서, 예시 풍부함은 단순 편의가 아니라 **속도·정확성의 직접 요인**이다.

## 단서 (언제 Next.js가 기본이 아닌가)

- 순수 정적 사이트/문서 → Astro 등이 더 가벼울 수 있음.
- 무거운 실시간/엣지 워크로드, 비-React 선호 팀 → 케이스별 재검토.
- 모바일 앱이 주력이면 다른 스택.

기본값은 "고민을 줄이는 출발점"이지 "절대 규칙"이 아니다.
