# 왜 Next.js를 기본 스택으로 쓰는가

> [!summary] 한 줄 요약
> 새 웹 프로젝트의 프레임워크를 매번 고민하지 않기 위해 Next.js를 "기본값"으로 둔다. 풀스택 적합성·예시 풍부함, 그리고 결정적으로 코딩 에이전트가 잘 다루는 스택이라는 점이 근거다.

## 핵심 주장

- **기본값으로 고민을 줄인다.** 특별한 이유가 없는 한 Next.js로 시작한다. 기본값은 "절대 규칙"이 아니라 "고민을 줄이는 출발점"이다.
- **축 1 — 풀스택 적합성.** Next.js는 프론트엔드(React UI)와 백엔드(API Routes / Route Handlers, Server Actions, RSC 데이터 페칭)를 한 코드베이스에서 다룬다. 작은 팀·1인 개발에서 프론트/백엔드 레포를 따로 운영하는 부담을 줄이고, 필요해지면 백엔드를 분리하는 점진적 경로가 자연스럽다.
- **축 2 — 예시 풍부함.** React 생태계의 사실상 표준 메타프레임워크라 문서·튜토리얼·오픈소스 예제·SO 답변·블로그가 압도적으로 많다. 인증(NextAuth/Auth.js), 결제(Stripe), DB(Prisma/Drizzle), 배포(Vercel) 등 흔한 조합마다 검증된 예시가 있다.
- **축 3 (핵심 논거) — 코딩 에이전트 시대에 예시 풍부함은 결정적 장점.** 에이전트는 학습 데이터·웹 검색·기존 예시 패턴에 기대어 코드를 생성한다. 예시가 풍부한 프레임워크일수록 더 정확하고 관용적인 코드를 내고, 니치한 프레임워크는 자료 부족으로 환각하거나 구식 API를 쓰기 쉽다. 따라서 "사람에게 익숙한 프레임워크"를 넘어 "에이전트가 잘 다루는 프레임워크"가 새 선택 축이 되었다.
- **단서 (언제 기본이 아닌가).** 순수 정적 사이트/문서(Astro 등이 더 가벼움), 무거운 실시간/엣지 워크로드나 비-React 선호 팀, 모바일 앱 주력이면 케이스별 재검토.

## 인용 가능 발췌

> "예시가 풍부한 프레임워크일수록 에이전트가 더 정확하고 관용적인(idiomatic) 코드를 낸다. 흔한 패턴일수록 에이전트가 헛디딜 확률이 낮다." — §3

> "'사람에게 익숙한 프레임워크'를 넘어 '에이전트가 잘 다루는 프레임워크'가 새 선택 축이 되었고, Next.js는 이 축에서 가장 앞선다." — §3

> "기본값은 '고민을 줄이는 출발점'이지 '절대 규칙'이 아니다." — 단서
