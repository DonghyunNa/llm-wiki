# 3대 연산 워크플로우

ingest / query / lint 각각의 정밀 절차. 디스패처(SKILL.md)에서 의도가 라우팅된 뒤 해당 에이전트가 따른다.

## Ingest

**목표:** 새 소스 1건을 위키에 누적. 단순 요약이 아니라 횡적 연결까지.

### 절차

1. **소스 식별·로드**
   - 입력이 `raw/`의 파일 경로면 그 파일을 읽는다.
   - 입력이 URL이고 `raw/`에 매칭 파일이 없으면 사용자에게 "클립 먼저 부탁드립니다" 안내 후 중단.
   - 입력이 본문 텍스트면 `raw/inbox/{YYYY-MM-DD}-{slug}.md`로 저장한 뒤 처리.

2. **사용자와 핵심 confirm**
   - 한 문단 요약 + 핵심 주장 3~5개를 보여주고 "이 방향 맞나요? 강조할 게 있나요?" 한 번 묻는다.
   - 사용자가 OK 하면 다음 단계. 수정 의견 있으면 반영.

3. **소스 페이지 작성**
   - `wiki/sources/{slug}.md`에 frontmatter + 요약 + 핵심 발췌 + 영향받을 후보 페이지 목록.

4. **횡적 갱신 (가장 중요)**
   - 영향받는 entity·concept 페이지를 식별 (위키 본문에 등장하거나 등장해야 할 명사).
   - 각 페이지에 대해:
     - 이미 있으면: 본문에 새 정보 추가 + frontmatter `sources`에 새 소스 wikilink 추가 + `updated` 갱신.
     - 없으면: 새 페이지 생성 (Entity/Concept 형식).
     - 기존 주장과 충돌하면: `> [!note] 충돌` 블록으로 양쪽 보존.
   - 사용자에게 한 번 더 confirm 권장 (여러 페이지를 한 번에 건드릴 때).

5. **index 등록**
   - 새/이름이 바뀐 페이지를 `wiki/index.md` 적절한 카테고리에 추가.
   - 기존 페이지가 의미상 크게 바뀌었으면 한 줄 요약도 갱신.

6. **log 기록**
   - `wiki/log.md` 끝에 `## [YYYY-MM-DD] ingest | {title}` 항목 추가.
   - 영향 페이지 wikilink로 나열.

7. **사용자 보고**
   - 한 줄: "{N}페이지 생성/갱신, index/log 동기화 완료." 영향 페이지를 wikilink로.

### 배치 ingest (여러 소스 한 번에 요청 시)

사용자가 "오늘 추가된 것들 ingest해줘"처럼 **여러 소스를 한 번에** 요청해도, 처리는 **반드시 직렬**이다 (근거: [[concurrent-ingest-race-condition]]).

1. 대상 소스를 모은다 (raw/inbox 등). 수집은 동시여도 무방 — race 없음.
2. **`date`(또는 파일명 날짜) 오름차순으로 정렬.** 번복 관계(`supersedes`)가 올바르려면 과거 결정이 먼저 vault에 반영돼야 함.
3. **1건씩 순차 처리.** 한 건의 횡적 갱신(index/log 포함)이 끝나야 다음 건 시작. 절대 동시/뭉뚱그려 처리 금지.
4. 한 건 실패 → 그 건만 skip + 보고에 표시, 배치는 계속 (실패 격리).
5. 보고: "N건 중 M건 성공, K건 skip" + 각 건 영향 페이지.

**자동/헤드리스 모드** (스케줄 배치 등 사람 없는 환경): confirm 단계를 건너뛰되, 결과를 git commit/PR로 올려 사람이 사후 검수. confirm을 *없애는* 게 아니라 *대화형 → 사후 PR*로 이동.

### 주의

- 한 ingest가 5~15페이지를 건드리는 게 정상이다. 너무 적으면 횡적 연결을 놓친 것.
- 그러나 너무 많은 사변적 페이지 생성은 금지 — 사용자에게 명시적으로 등장하지 않은 개념까지 미리 만들지 않는다.
- 사용자와의 대화 자체가 가치 있는 메타 정보다. 강조점·관심사·반응을 소스 페이지의 `## 메모`에 한두 줄 남긴다.

## Query

**목표:** 위키를 자료로 질문에 답한다. 답이 가치 있으면 위키로 환류.

### 절차

1. **검색 범위 좁히기**
   - `wiki/index.md` 읽기 → 카테고리·요약으로 후보 페이지 식별.
   - 필요하면 `grep -rn "키워드" wiki/`로 본문 매칭 보강.
   - 후보가 너무 많으면 frontmatter `tags`로 필터.

2. **본문 읽기**
   - 후보 페이지를 실제로 read. 각 페이지의 frontmatter `sources`도 따라가 필요 시 source 페이지 본문까지.

3. **종합 + 형식 결정**
   - 질문에 따라 형식 선택:
     - 사실 한 줄 → 한 줄 답.
     - 비교/대조 → 마크다운 표.
     - 진척/변화 → 타임라인.
     - 큰 분석 → 헤딩 있는 본문.
   - 모든 주장에 인용 `(→ [[page]])` 또는 `[1]` 첨부.

4. **gap 명시**
   - 질문의 일부에 위키 근거가 없으면 명시.
   - 외부 보강 권고 또는 새 소스 ingest 권고.

5. **답 표시 + file-back 제안**
   - 답을 사용자에게 보여준다.
   - 충분히 가치 있는 답이면(비교표, 분석, 새로운 연결) "이걸 `wiki/syntheses/{slug}.md`로 저장할까요?" 한 번 묻는다.
   - 사용자가 OK → 페이지 생성 + index/log 동기화.
   - 사용자가 거절 → 그냥 둠. 같은 세션에서 다시 안 묻는다.

### 주의

- vault 전 페이지 grep은 큰 vault에서 비싸다. 항상 index부터.
- 환각 금지. 출처 없는 주장은 답에 넣지 않는다.
- 답이 짧으면 file-back 제안 안 해도 된다. 보통 비교/분석/연결 발견 같은 경우만 가치 있음.

## Lint

**목표:** 위키 정합성·건강을 진단. **수정 없이** 우선순위 리포트만.

### 절차

1. **범위 결정**
   - 사용자가 범위 지정 (`전체`, `최근 N일`, 디렉토리) → 따른다.
   - 기본값: 전체.

2. **6개 카테고리 검사** (순서대로)
   1. **모순**: 닫히지 않은 `> [!note] 충돌` 블록 + 같은 사실에 대한 상반된 주장.
   2. **Orphan**: 다른 페이지에서 wikilink로 참조 안 되고 index에도 없는 페이지. frontmatter `orphan: true`는 제외.
   3. **Stale claim**: entity/concept 페이지의 cited sources 중 가장 최근 것이 페이지 `updated`보다 더 새로움 → 갱신 후보.
   4. **Missing cross-reference**: 페이지 A 본문에 페이지 B의 표준 명칭(또는 alias)이 등장하지만 `[[B]]` 위키링크가 없음.
   5. **Concept-without-page**: 본문에 N회 이상 등장하지만 페이지 없는 명사. vault 크기에 따라 임계치 조절.
   6. **Data gap**: 페이지의 `## 열린 질문` 블록 + sources 0~1개인 핵심 페이지.

3. **우선순위 매기기**
   - P0: 모순 (정합성 깨짐). 즉시.
   - P1: stale + 핵심 missing cross-reference. 이번 주.
   - P2: orphan, concept-without-page, data gap. 시간 날 때.

4. **리포트 표시**
   - 사용자에게 마크다운 형식으로 표시 (파일 저장 안 함).
   - 다음 행동 권고 포함 (어떤 ingest를 하면 gap이 메워질지 등).

5. **로그 1줄**
   - `wiki/log.md`에 `## [YYYY-MM-DD] lint | P0:N P1:M P2:K` 추가.

### 주의

- **절대 자동 수정 금지.** lint는 진단만. 수정은 사용자 결정.
- 깨끗하면 깨끗하다고 보고. 억지로 issue 만들지 않는다.
- 단순 휴리스틱(grep, file metadata)으로 잡히는 것부터. AI 추론은 마지막에.

---

## 운영 팁

- **ingest 한 번에 너무 많은 소스 처리 금지.** 1건씩, 사용자와 confirm하며 진행 권장.
- **query 답을 file-back할지는 가치 기준으로.** 단순 사실 답은 파일링하지 않는다.
- **lint는 주기적으로.** "이번 주 lint" 정도로 정기화 권장.
- **log를 사람도 읽는다.** vault의 역사·궤적 자체가 자료.
